No agent code, no LLM Gateway, no audit chain, no RAG, no migrations. Those are Phases 1–4. Phase 0 is purely scaffold.

### Local dev database (`scripts/`)

This is the load-bearing change in the updated prompt. No Docker; native Postgres only.

- `scripts/setup-dev-db.sh` — executable bash script (`#!/usr/bin/env bash`, `set -euo pipefail`). Idempotent. Defensive pattern applied:
  1. **Sanitise:** read DB name from env (`DEV_DB_NAME`, default `agentic_claims_dev`). Trim, lowercase. Require it to match `^[a-z][a-z0-9_]*$`.
  2. **Validate:** confirm `psql` is on PATH; confirm a Postgres server is reachable (`pg_isready`); confirm version >= 16 by parsing `psql --version`; confirm the `vector` extension is available (`SELECT 1 FROM pg_available_extensions WHERE name='vector'`).
  3. **Abort:** on any check failure, print a diagnostic message naming what was expected vs found and pointing at the README install instructions; exit non-zero. No silent skips.
  4. **Execute:** create the database if it doesn't exist (`createdb` with idempotent guard); connect and run `CREATE EXTENSION IF NOT EXISTS vector`; print a final confirmation listing the database name, server version, and pgvector version.

  The script does not manage roles or passwords — local Postgres.app and `brew install postgresql@16` both grant the OS user superuser by default, which is the lowest-friction local setup. Phase 1 layers role and connection-string handling on top of this when there's something to connect.

- `scripts/.gitkeep` is unnecessary because `setup-dev-db.sh` itself populates the directory.

## Approach and key design decisions

1. **Top-level `pyproject.toml` rather than `backend/pyproject.toml`.** `uv init` produces a single project at the repo root by default. A single project simplifies CI, locking, and editor tooling. The Python source still lives under `backend/` as a package; only the project metadata sits at root. The "Project Structure (target)" comment in `CLAUDE.md` shows `backend/pyproject.toml` with a parenthetical "(or top-level)", so the top-level placement is already sanctioned. Flagging explicitly so it's an informed choice.

2. **Health endpoint at root, not under `/api`.** Render's health checks and most container platforms expect a fixed path; keeping it out of the versioned `/api` prefix avoids a future breaking change. The placeholder `/api` router is mounted but empty.

3. **Settings loader strategy.** Pydantic Settings handles env vars natively. For the YAML layer I add a thin `_load_yaml_overrides()` helper invoked inside `Settings`'s `model_validator(mode="before")`. The defensive pattern applies: sanitise the YAML path → validate the file is readable → abort with a typed exception if the YAML is malformed → execute. Phase 0 only has flat keys, so the helper is small; Phase 1 will extend it for nested sub-models. The malformed-YAML guard gets a triggering test that asserts the error message contains the path and the underlying YAML error excerpt.

4. **CORS configuration.** Once deployed, the Vercel frontend hits the Render backend cross-origin. I'll wire CORS middleware reading allowed origins from `Settings.cors_allowed_origins` (default `["http://localhost:5173"]` for dev). This locks the cross-origin pattern early and avoids a Phase-7 retrofit. Flagging because it crosses a boundary.

5. **`/health` version field.** Decision: read from `pyproject.toml` via `importlib.metadata.version("agentic-claims-poc")`. Three lines, keeps the field truthful, and means future version bumps don't drift. The project metadata sits at the repo root with `name = "agentic-claims-poc"` so the lookup resolves once `uv sync` installs the project in editable mode.

6. **`uv` workflow.** I will run `uv init` at the repo root (the existing `README.md` is fine; `uv init` doesn't overwrite it), then `uv add` and `uv add --dev` for each declared dependency, then `uv sync` to generate the lockfile. If any flags don't match the local `uv` version (0.9.11), I'll adjust and note it in the report.

7. **Frontend bootstrap via `npm create vite@latest`.** Standard, current path. Tailwind v4 setup follows the Vite plugin guide. I'll trim the generated boilerplate to the minimum that demonstrates Tailwind is wired (one Tailwind utility class visible in the rendered output), so the app isn't carrying example logo/counter code into Phase 1.

8. **`scripts/setup-dev-db.sh` is bash, not Python.** Three reasons: (a) it runs before the Python project is necessarily synced; (b) the operations are all shell-native (`psql`, `createdb`, `pg_isready`); (c) keeping it standalone avoids a chicken-and-egg dependency on the Python venv. The script does follow the same defensive ordering (sanitise → validate → abort → execute) — it just uses bash idioms.

9. **No `.env.example` at the root in Phase 0.** Backend has no secrets yet. `frontend/.env.example` documents the only Phase 0 env var (`VITE_API_BASE_URL`) and establishes the convention. Phase 1 introduces the backend `.env.example` when `DATABASE_URL` and the LLM API keys arrive.

10. **Anonymisation check.** I will grep the staged files for the client name before commit. The current repo content is already clean, but the check is cheap insurance.

## Risks and downstream impacts

1. **Top-level vs nested `pyproject.toml`.** Phase 1 onwards assumes the layout I pick here. Switching later means moving the project file and updating CI paths — small but real churn. Confirming the top-level choice now avoids a Phase-1 rework.

2. **YAML loader contract.** Whatever shape `_load_yaml_overrides` settles on in Phase 0 is the shape Phase 1's nested sub-models will extend. I'll keep it deliberately simple (top-level keys merged into the default model) so nesting doesn't require a rewrite. Flagging this as the first interface that crosses a boundary — Phase 1's settings expansion will need to honour this contract.

3. **Native-Postgres assumptions on contributor machines.** The setup script assumes `psql` on PATH and a running local server. If a future contributor uses Linux or Windows, the script's macOS-flavoured guidance won't fully apply (Linux path: `apt install postgresql-16 postgresql-16-pgvector`). For a single-developer prototype this is fine; flagging in case the audience expands. I will add a one-liner in the README noting macOS is the supported dev platform.

4. **Render free-tier cold starts.** Render's free Web Service tier sleeps after inactivity and takes ~30s to wake. Health checks during the demo may see a cold start. Acceptable for a prototype; documented in Phase 7's README as a known constraint. Flagging now so it isn't a surprise later.

5. **Vercel + Render CORS.** Already addressed (point 4 in "Approach"). Flagging again here because it's an early decision that locks the cross-origin pattern.

6. **CI runtime cost.** Two-job workflow + uv sync + npm ci on every PR is fast (under 2 minutes) but worth flagging that we'll consume free-tier Actions minutes from day one.

7. **No tests for code that doesn't exist yet.** The build-log will record one health test + one settings test + one YAML-guard test on the backend, and one render test on the frontend. This is the spec; calling it out so the test count in the build-log entry isn't a surprise.

## Interface stability — what crosses a boundary in Phase 0

- **`/health` JSON shape.** Once deployed, monitoring and the frontend depend on it. Locked as `{"status": "ok", "version": str}`. Both fields stable.
- **`Settings` model field set.** Phase 1 extends it with sub-models; Phase 0 fields stay where they are.
- **`settings.yaml.template` schema.** Same stability commitment as `Settings`.
- **CI workflow file path and job names.** External signals (status checks on PRs) will start referencing these.
- **`scripts/setup-dev-db.sh` contract.** The default DB name (`agentic_claims_dev`), the `DEV_DB_NAME` env var, and the script's exit-code semantics (0 on success, non-zero on any guard failure) become the contract Phase 1's migrations and connection settings rely on.

Anything that would break those gets flagged in a future plan before changing.
